# AIコンシェルジュ主導型サイト — 設計ドキュメント

最終更新: 2026-07-10  
関連: [`SITE_EXPERIENCE_REDESIGN.md`](./SITE_EXPERIENCE_REDESIGN.md) / 実装: [`AI_CONCIERGE_IMPLEMENTATION_PLAN.md`](./AI_CONCIERGE_IMPLEMENTATION_PLAN.md)

## 0. この文書の目的

ideal 公式サイトの AIコンシェルジュを、

- **右下に常駐する問い合わせチャット**

から、

- **ページ文脈を理解し、案内・要件整理・概算まで導くサイト横断の案内役**

へ育てるための設計方針。

本機能自体が、サービスとしての **「AIコンシェルジュ型Webサイト」の自社デモ兼営業導線** になる。

サイト体験再設計（DEMOS → CASES → SERVICES → CONTACT）の延長線上に位置づける。

---

## 1. 背景と現状

### すでにできていること（Phase 0）

現行実装は **LLM 不使用の Multi-step Concierge**。

```
相談目的を選ぶ
 → 現在の状況を選ぶ
 → 開始時期を選ぶ
 → 選択内容を整理（Markdown）
 → お問い合わせへ引き継ぐ
```

| 資産 | 場所 |
|------|------|
| 選択フロー定義 | `lib/concierge/ideal-flow.ts` |
| フローUI | `components/concierge/IdealConciergeFlow.tsx` |
| 完了画面 | `components/concierge/ConciergeDoneStep.tsx` |
| 開閉・serviceHint | `components/concierge/concierge-context.tsx` |
| Contact 引き継ぎ | `lib/concierge/contact-prefill.ts` |
| サービスからの起動 | `OpenConciergeButton` + `serviceHint` |

入口の選択式UIは、小規模事業者向けに「何を書けばいいか分からない」問題を解消しており、**残す前提**。

### まだ足りないこと

| ギャップ | 内容 |
|----------|------|
| ページ文脈 | FAB は文脈なし。`serviceHint` はサービスIDのみ |
| 案内 | 関連デモ・Cases・記事への動的誘導がない |
| AI接続 | 自由入力の解釈・追加質問・要件ドラフトがない |
| 結果UI | 完了画面が Markdown テキスト中心 |
| 概算 | 価格ルールも算出UIもない |

## 2. 目指す体験

### 理想導線

```
サイト訪問
 → Hero / デモを見る
 → 「これ、自社でも使えるかな？」
 → AIコンシェルジュを開く（今見ているページを理解している）
 → 3〜5問程度のヒアリング（選択 + 必要なら自由入力）
 → 自社への応用案・簡易要件定義を生成
 → 概算費用（ルールベース）
 → 問い合わせに内容を引き継ぐ
```

通常の制作会社サイト:

```
サービス → 実績 → 問い合わせフォーム（名前・会社・本文）
```

ideal:

```
デモ体験 → 自社への応用をAIと考える → 要件整理 → 概算 → 人に相談
```

サイト訪問自体が **簡易コンサルティング体験** になる。

### 3つの役割

| # | 役割 | やること | やらないこと |
|---|------|----------|--------------|
| 1 | **案内する** | 課題・閲覧ページから関連デモ / Cases / サービスへ誘導 | サイト内全文検索の代替を目指しすぎない |
| 2 | **要件を整理する** | 課題・利用者・想定機能・進め方を構造化して見せる | 契約レベルの正式要件定義書を自動確定しない |
| 3 | **概算を出す** | ルール表に当てはめた参考レンジを表示 | AIに金額を自由生成させない |

### ハイブリッド対話（入口方針）

```
入口は選択式（現行を活かす）
 → 必要に応じて自由入力
 → AIが追加質問（少数）
 → 要件整理UI
 → 概算（任意）
 → Contact 引き継ぎ
```

最初から全部を自由チャットにしない。選択式は「書けない人」の入口として維持する。

---

## 3. 設計原則

1. **文脈ファースト** — 開いた瞬間の第一声は、今見ているページから決まる
2. **選択式を捨てない** — AIは選択の後段・自由入力の解釈に使う
3. **金額はルールベース** — AIは分類と説明、計算は価格表
4. **結果はプロダクトUI** — Markdownの壁ではなく、状況 / 方向性 / 機能 / デモ / 次のステップ
5. **問い合わせがゴール** — コンシェルジュ完了 = Contact への構造化引き継ぎ
6. **自社デモである** — この導線自体を「AIコンシェルジュ型サイト」の証拠にする
7. **サイト体験IAと一体** — DEMOS / CASES / SERVICES / LAB の次の一手を出す存在

## 4. ページ文脈モデル

コンシェルジュは「チャット履歴」だけでなく、起動時の **PageContext** を持つ。

### 概念型

```ts
type ConciergePageType =
  | 'home'
  | 'demo'
  | 'demo_hub'
  | 'service'
  | 'case'
  | 'lab'
  | 'insight'
  | 'contact'
  | 'other'

type ConciergePageContext = {
  pathname: string
  pageType: ConciergePageType
  /** デモ slug（例: voice-to-structured） */
  demoId?: string
  /** Cases slug（例: construction-photo-sorting） */
  caseSlug?: string
  /** サービスID（例: ai-consulting）— 現行 serviceHint と互換 */
  serviceId?: string
  /** Insights slug 等 */
  insightSlug?: string
  /** 業界タグ（建設・介護など） */
  industry?: string
}
```

### ページ種別ごとの第一声（例）

| 閲覧中 | 第一声の意図 | 初期選択肢の例 |
|--------|--------------|----------------|
| デモ詳細（音声→構造化 等） | この仕組みの自社応用を提案 | 自社業務で使えるか相談 / 類似事例 / 機能整理 / 概算 |
| デモ Hub | どの変化に近いかを聞く | 7変化への誘導 or 課題から選ぶ |
| Webサービス | Web制作の状況整理 | 新規 / リニューアル / LP / 業務システム / 未定 |
| AIサービス | 業務課題のヒアリング | 効率化 / プロダクト組込 / データ活用 / 未定 |
| Cases（建設×写真 等） | 同業界・同課題の相談を提案 | 現場写真 / 日報 / 報告書 / 音声 / その他 |
| LAB / Insights | 深掘りから実務相談へ戻す | 関連デモへ / 相談を始める |
| その他・TOP | 現行の root 選択フロー | Web / アプリ / AI / BC / まだ分からない |

pathname からの解決は、実装PLANの **Context Registry**（ルール表）で行う。LLMに「今どのページか」を推測させない。

## 5. 会話〜成果物の情報設計

### 5-1. セッション状態（概念）

```
PageContext
 + IdealTrack（web / app / ai / bc / unsure）
 + 選択回答（situation / timeline 等）
 + 自由入力（任意）
 + AI追加Q&A（任意）
 → ConciergeResult（構造化）
 → Estimate（任意・ルール）
 → ContactPrefill
```

### 5-2. ConciergeResult（完了画面の中身）

Markdown ではなく、少なくとも次のブロックを持つ。

| ブロック | 内容 |
|----------|------|
| 現在の状況 | 選択・ヒアリングから要約 |
| 希望時期 | timeline 等 |
| おすすめの方向性 | 短い提案文 + カテゴリ |
| 想定機能 | タグ列（例: アップロード / AI分類 / 管理画面） |
| 関連デモ / Cases | カード1〜2 + リンク |
| 次のステップ | 要件を深める / 概算 / この内容で相談 |

### 5-3. 概算（Estimate）

**方針: 金額計算はルールベース、説明と質問はAI。**

内部に価格ルール（例）を持つ。

| 項目 | 参考レンジ（例・要調整） |
|------|--------------------------|
| LP制作 | 15〜30万円 |
| コーポレート | 30〜80万円 |
| 小規模Webアプリ | 50〜150万円 |
| AIプロトタイプ | 30〜100万円 |
| OCR追加 | ＋10〜30万円 |
| 外部API連携 | ＋10〜50万円 |
| 管理画面 | ＋20〜60万円 |

フロー:

1. 相談内容から **適用する行項目** を選ぶ（ルール or AI分類→ルール照合）
2. レンジを合算して表示（例: 80〜150万円程度）
3. 必ず注記: 「現時点の情報から算出した参考価格。データ量・精度・連携・管理機能で変動」

AIに「いくらかかりますか？」と自由回答させない。

## 6. サイト全体との接続

[`SITE_EXPERIENCE_REDESIGN.md`](./SITE_EXPERIENCE_REDESIGN.md) の4分類に対するコンシェルジュの役割。

| 種別 | ユーザーの問い | コンシェルジュの仕事 |
|------|----------------|----------------------|
| DEMOS | 何ができる？ | 「自社でも？」へ転換し、応用ヒアリングへ |
| CASES | うちでも使える？ | 同型課題の整理・関連デモ・相談へ |
| SERVICES | 何を頼める？ | 状況選択→要件→概算→Contact |
| LAB | なぜ・どう作る？ | 実務導線（デモ/サービス/相談）へ戻す |

中心の体験ループ:

```
見る → 触る → 自社に置き換える → AIに相談する
 → 要件を整理する → 概算を見る → 人に相談する
```

## 7. プロダクトとしての位置づけ

ideal が売るものの一つとして、

**AIコンシェルジュ型Webサイト**

を説明できる状態にする。

| 従来サイト | AIコンシェルジュ型 |
|------------|-------------------|
| ユーザーが情報を探す | ユーザーが目的を話す |
| ページを読む | AIが状況を整理する |
| 問い合わせする | 関連体験へ案内し、必要情報を集めて見積・相談へ |

営業時は「作れます」ではなく **「うちのサイトで実際に使ってみてください」** と言えることがゴール。

応用領域の例: 工務店、不動産、士業、採用、EC、医療・介護、教育、M&A、BtoB 等。

## 8. スコープ境界

### やる（本構想の範囲）

- ページ文脈オープン（第一声・初期選択肢）
- 選択式 + 自由入力ハイブリッド
- 構造化結果UI
- ルールベース概算
- Contact への構造化引き継ぎ
- 関連デモ / Cases への案内

### やらない（当面）

- 全ページの自由対話型カスタマーサポート（FAQ全置換）
- 正式見積書・契約の自動発行
- ユーザーアカウント・会話履歴の長期保存（初期はセッション単位で可）
- サイト内のあらゆるコンテンツをLLMに丸投げ検索

### リスクと対策

| リスク | 対策 |
|--------|------|
| よくあるチャットボットに見える | 文脈第一声と結果UIを必須にする |
| 金額の誤案内 | ルール表のみ。注記必須。AIは金額生成禁止 |
| 範囲爆発 | 文脈オープン → 結果UI → AI → 概算の順を守る |
| 選択式を捨てて離脱増 | root / 状況選択は維持 |

---

## 9. 成功の定義

1. デモページで開くと、そのデモ名が第一声に出る
2. 完了画面が「整理された相談カード」として読める
3. 問い合わせフォームに、状況・方向性・（あれば）概算が引き継がれる
4. 社内・営業が「コンシェルジュ型サイトのデモ」として案内できる
