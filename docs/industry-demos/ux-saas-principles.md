# 業種デモ UX 原則（製造の学び → 横断）

最終更新: 2026-07-24  
適用: ideal 業界本線の体験デモ（製造・建設・医療福祉・小売／飲食 ほか）  
関連: [`README.md`](./README.md)（3本建て IA）／製造プレイヤー正本: [`product_flow/docs/experience-player-ux.md`](../../../../product_flow/docs/experience-player-ux.md)

---

## 0. いちばん大事な約束

**業種ごとに UI／ビジュアル言語は必ず別物にする。**

- 製造のシェル・コンポーネント・配色・タイポを、他業種へコピーしない
- 「同じ体験の型（IA）」と「同じ見た目」を混同しない
- 業種を見れば、ロゴを消しても「どの現場向けか」が分かること

共通にしてよいのは **体験の設計ルール**（短い・一点・サンプル先行）だけ。  
見た目・操作感・情報密度・端末前提は業種専用。

---

## 1. 目的（全業種共通）

「説明されて理解するデモ」ではなく、**触って使いたくなるデモ**にする。

参考にするのは業界特化の古い業務画面ではなく、**現代の魅力的な SaaS／スマホアプリ**の:

- 最初の画面で主操作が分かる
- 余白と階層で落ち着いて読める
- モーションは気配（ノイズにしない）
- 完了感がはっきりする

業種ごとに「どの SaaS／アプリ群を参考にするか」は §4 で固定する。

---

## 2. 体験の型（共通・見た目は共通にしない）

製造で効いた **構造**だけを横断する。スキンは業種専用。

| ルール | 内容 |
|--------|------|
| 5秒ルール | 説明文なしで「次に押すもの」が分かる |
| 1画面1仕事 | ハブ＝選ぶ／プレイヤー＝そのテーマを完走 |
| 1デモ1クライマックス | 感動ポイントは質問または場面が1つ |
| サンプル先行 | AI接続・アップロードはツアー後の任意 |
| 状態はUIで | チップ・ロック・完了表示。長文の「使い方」禁止 |
| 次の一歩は副次 | ROI・相談は折りたたみ／右寄せ等。主役を奪わない |
| 短時間 | ガイド数分。WF全体再現はしない（[`README.md`](./README.md)） |

製造プレイヤーの画面骨格は **製造専用の一例**であり、他業種のワイヤーにしない。

---

## 3. 禁止（見た目のコピペ）

| 禁止 | 理由 |
|------|------|
| 製造の `PlayShell` / StepRail / AnswerCard を他業種に移植して色だけ変える | 「別デモ」に見えない |
| 全業種で同じ紺＋グレーの業務UI | ideal の業界カードが死ぬ |
| デスクトップ前提の密なテーブルを、現場スマホ業種にそのまま出す | 使う気が削げる |
| 業種横断の「共通デモデザインシステム」を先に作り、後から業種色を乗せる | 原則0に反する |

許容: ideal **公式サイト**側のナビ・フッター・相談フォームなどサイト殻の共有。  
**体験デモ本体のキャンバスは業種専用。**

---

## 4. 業種別ビジュアル方針（必須）

各業界 PLAN に「ビジュアル方針」節を置き、ここを正本参照する。実装前に色・タイポ・端末・参考アプリを1行でも固定する。

### 4.1 製造（先行・確定）

| 項目 | 方針 |
|------|------|
| トーン | **最低限のシンプル**。文書と根拠が主役 |
| 色 | 抑えめ（紺／ニュートラル）。装飾少なめ |
| 文字 | 読みやすさ優先。煽らない |
| 端末 | デスク＋現場タブレット両対応。過剰なモバイル演出はしない |
| 参考 | Linear / Notion 的な「静かで速い業務ツール」 |
| 正本UI | [`experience-player-ux.md`](../../../../product_flow/docs/experience-player-ux.md) |

製造は「地味でよい」。他業種が派手でも、製造を無理に派手にしない。

### 4.2 建設・設備

| 項目 | 方針 |
|------|------|
| トーン | **屋外・手袋でも押せる**。目立った色、大きな文字 |
| 色 | 高コントラスト。警告・完了が遠くから分かるアクセント |
| 文字 | 大きめ。ラベル短く。一文は切る |
| 端末 | **スマホメイン前提**。片手・日光下でも読める密度 |
| 参考 | 優れた現場系／配送・工事系モバイルアプリ、大きなタップターゲットのコンシューマアプリ |
| やらないこと | 製造プレイヤーの細いステップレールの流用、PC密テーブルをそのまま |

### 4.3 医療・福祉（介護含む）

| 項目 | 方針 |
|------|------|
| トーン | **白基調**。清潔・安心・誤操作しにくい |
| 色 | 白／ごく薄いニュートラル。アクセントは医療・ケアで信頼できる落ち着いた色1つ |
| 文字 | 明瞭。急かさない。重要操作は確認しやすい |
| 端末 | 記録端末・タブレット多め。騒がしいアニメ禁止 |
| 参考 | 優れたヘルスケア／記録系アプリの「余白と安心感」 |
| やらないこと | 建設のような強コントラスト派手UI、飲食のようなブランド過剰演出 |

### 4.4 小売・サービス（店舗オペ）

| 項目 | 方針 |
|------|------|
| トーン | 店舗のテンポに合う。応対が止まらない感じ |
| 色 | 業態に合わせてよいが、製造の地味UI・医療の白基調とは明確に別 |
| 文字 | 客前でも一瞬で読める短さ |
| 端末 | スマホ／ハンディ想定可 |
| 参考 | 優れた店舗オペ・サポート系 SaaS |

### 4.5 飲食（小売ハブ外でも同じ原則）

| 項目 | 方針 |
|------|------|
| トーン | **UIUXデザインを特に意識**。ブランド感・食欲・忙しさの両立 |
| 色 | 業態らしい魅力的なパレット。テンプレ業務画面に見えないこと |
| 文字 | 表情のあるタイポ（読みやすさは維持） |
| 端末 | 店舗・バックヤードのスマホ／タブレット |
| 参考 | 優れた飲食・予約・シフト系コンシューマ級アプリ |
| やらないこと | 「業務デモあるある」のグレーカードだらけ |

## 5. 業種を始めるときのチェック（デザイン）

実装に入る前に、その業種 PLAN または短いメモで次を埋める。

1. **参考にする実在アプリ／SaaS を2〜3**（名前でよい）
2. **主色・背景・アクセント**（製造と被らないこと）
3. **主端末**（スマホ／タブ／デスク）
4. **文字の大きさの方針**（建設＝大、製造＝標準、など）
5. **ロゴを隠したスクショで業種が分かるか**（セルフテスト）

満たせないなら、製造シェルの流用を疑う。

---

## 6. 受け入れ（横断）

### 体験（全業種）

- [ ] 5秒で主操作が分かる
- [ ] 1テーマのクライマックスが言える
- [ ] サンプルだけで完走できる（AIは任意）
- [ ] ROI／相談が主役を奪っていない

### 見た目（業種ごと）

- [ ] 製造デモと並べて **別プロダクト** に見える
- [ ] §4 のその業種方針（色・文字・端末）に沿っている
- [ ] 参考にした魅力的 SaaS／アプリの「使いたさ」が触感に落ちている
- [ ] （建設）スマホ幅でタップ・可読が破綻しない
- [ ] （医療福祉）白基調で落ち着いている
- [ ] （飲食）デザイン意識が伝わる（業務グレー一色でない）
