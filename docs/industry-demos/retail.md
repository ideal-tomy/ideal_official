## 1. 楔（隙間）と証明する一文

### 楔

**定型の顧客応対** と **スタッフの裏方判断（ルール・在庫感覚・次の一手）** が、人の記憶と個別対応に張り付いていること。  
営業時間外・混雑時に案内が止まり、属人的な判断で品質がぶれる。

### 証明する一文

顧客への回答、スタッフの根拠確認、数字に基づく次の判断が、それぞれ短い体験で実感でき、つながると店舗オペの切れ目が減る。

---

## 2. ハブ構成

組み合わせ文（ハブに必ず出す）:

> 顧客に答える → スタッフが根拠で揃える → 数字で次を決める。つながると「応対も判断も、人の勘だけに依存しない店舗」になる。

| # | Demo ID | テーマ名 | 体験の核（一言） | 主役役割 | status |
|---|---------|----------|------------------|----------|--------|
| ① | `ret-customer-chat` | 顧客応対 | FAQ・返品等のチャット案内 | 顧客視点・店舗責任者 | `ready_asset` |
| ② | `ret-staff-knowledge` | スタッフ判断 | 店舗ルール・マニュアルを根拠付きで | スタッフ・店長 | `needs_build` |
| ③ | `ret-demand-next` | 次の判断支援 | 需要・来客の予測と打ち手 | 店長・経営 | `needs_build` |

推奨の初回着地: **Demo①**（現行外部・完成度高）

**飲食シフト**は本3本に入れない。後節「隣接楔」を参照。

## 3. Demo 詳細

### Demo① — 顧客応対

- **Demo ID:** `ret-customer-chat`
- **テーマ:** 業種別の定型問い合わせに、その場で答え、必要なら有人へつなぐ
- **体験してもらうこと:** 小売コマース等のパックで、商品・配送・返品などの質問→根拠付き回答
- **感じてほしいこと:** 「毎回人が答えなくてよい定型がある。営業時間外も止めない」
- **主役役割:** 店舗・EC責任者（顧客体験の持ち主）
- **クライマックス:** 返品や配送などポリシー込みの質問に、根拠を示して答えられる
- **ガイド手数:** 3〜4（推奨質問クリック）
- **AIの使い方:** サンプル／BYOK／体験コード（既存 Access Mode）。ハブではサンプル推奨を先に
- **既存資産:**
  - リポジトリ: [`customer_support_demo`](../../../../customer_support_demo)
  - 公開: `https://customer-support-demo-lime.vercel.app/`
  - 業種パック: `docs/industry-packs/01_retail-commerce` ほか
  - 現行TOP小売カードの `tryHref`
- **status:** `ready_asset`
- **ギャップ:** ハブ①としての位置づけコピー。②③への誘導が未設計
- **実装ステップ:**
  1. ハブ①＝現行CSデモ（小売パック既定）
  2. 完了後「スタッフ側の同じルールを見る」→②
  3. Dual Lens（顧客×導入）は既存 docs をハブ説明に要約転記
- **受け入れ（Demo単体）:** 「定型問い合わせをチャットで止められる」と言える

### Demo② — スタッフ判断

- **Demo ID:** `ret-staff-knowledge`
- **テーマ:** 店頭・バックヤードのスタッフが、規程・マニュアルを根拠付きで確認する
- **体験してもらうこと:** 「この返品、今日受けてよいか」「会員ポイントの例外は」などスタッフ視点の質問 → 回答＋出典
- **感じてほしいこと:** 「顧客向けFAQの裏側を、スタッフも同じ根拠で揃えられる」
- **主役役割:** スタッフ・店長
- **クライマックス:** スタッフ向けの迷い質問に、出典付きで答え、属人回答が減るイメージが持てる
- **ガイド手数:** 4
- **AIの使い方:** サンプル先行。ナレッジ系はライブが効く
- **既存資産:**
  - ベース: [`internal_knowledge_demo`](../../../../internal_knowledge_demo)（現行は就業規則・社内寄りの想定が強い）
  - 公開: `https://internal-knowledge-demo.vercel.app/`
  - CS側ナレッジとの役割分担をハブで明示する必要あり
- **status:** `needs_build`
- **ギャップ:** 小売・店舗オペ向けサンプル文書と推奨質問への寄せ（HRデモのままでは小売ハブ②として弱い）
- **実装ステップ:**
  1. 小売スタッフ向けサンプル（返品権限、レジ例外、シフト申し送り級の店舗ルール）をパック化
  2. 推奨質問を「スタッフが同僚や店長に聞いていること」に限定
  3. ①の顧客質問と対になる「裏のルール」を1組以上用意（組み合わせ想像用）
  4. 公開入口を②専用コピーに（社内全般ナレッジと混同しない）
- **受け入れ（Demo単体）:** 「スタッフが根拠付きで判断できる」を自店に置き換えられる

### Demo③ — 次の判断支援

- **Demo ID:** `ret-demand-next`
- **テーマ:** 来客・需要の予測を、次の打ち手とセットで見る
- **体験してもらうこと:** 予測値・信頼区間・影響要因 → 「だから今日はこうする」レベルの示唆
- **感じてほしいこと:** 「感覚の発注・配置が、数字の仮説になる」
- **主役役割:** 店長・経営
- **クライマックス:** 予測だけでなく、要因と次アクション候補がセットで出る
- **ガイド手数:** 3〜4
- **AIの使い方:** ギャラリーはサンプル演出中心。本格化時は店舗シナリオの固定データで十分（過度なMLデモにしない）
- **既存資産:**
  - サイト内試食: `/ai-capability-gallery/data-to-prediction`
  - 事例: `retail-demand-prediction`（externalDemo なし）
- **status:** `needs_build`
- **ギャップ:** ギャラリーは簡易。小売ハブ③としては「店舗の一日」シナリオ・打ち手文言・SPでのグラフ露出など体験強化が必要。専用リポジトリは未必須（サイト内拡張または軽量デモでも可）
- **実装ステップ:**
  1. ③のクライマックスを「予測＋打ち手」に固定（予測グラフだけにしない）
  2. 小売シナリオ（週末来客、天候、販促）のサンプルに寄せる
  3. ハブ③カードから強化版へ。未完成期間はギャラリー＋「本格準備中」ではなく、最小の打ち手UIを足して `ready_asset` に近づける
  4. 事例 `retail-demand-prediction` に external／ハブリンクを後日追加
- **受け入れ（Demo単体）:** 「数字を見て次の一手が言える」体験だと分かる

## 6. Won't（やらないこと）

- 小売フル基幹（POS・在庫・会計）を1デモにする
- 飲食シフトを小売3本の同格にする
- ①のCSにスタッフ向けと顧客向けを混ぜてテーマをぼかす（②を別デモにする）
- ③で過剰な予測精度アピール（打ち手と意思決定支援が本番）
- 美容・飲食などCS業種パックを、ハブの4本目として並べる（①の中の切替に留める）
